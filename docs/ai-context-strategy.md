# AI context strategy

**Version:** 1.1.0 (DEC-014 multi-agent-safe layout)
**Date:** 2026-06-21
**Applies to:** ai-architect-6-chat (`ai-chat`)

## Purpose

Define how AI coding agents (Cursor, Claude Code, Copilot, etc.) consume project context in a
tool-agnostic way — and how parallel agents in separate worktrees avoid merge conflicts on memory
state.

## Layer model

```text
┌─────────────────────────────────────────────────────────┐
│  Optional IDE adapters (.cursor/, MCP, etc.) — future    │
├─────────────────────────────────────────────────────────┤
│  .agents/skills/<name>/SKILL.md   ← canonical skills     │
├─────────────────────────────────────────────────────────┤
│  .agents/memory-bank/             ← session continuity   │
│    ├ reference/registry/records/locks/worktrees          │
├─────────────────────────────────────────────────────────┤
│  Module AGENTS.md (2–5 files)     ← module boundaries    │
├─────────────────────────────────────────────────────────┤
│  AGENTS.md (root)                 ← compact index        │
├─────────────────────────────────────────────────────────┤
│  docs/*.md                        ← canonical deep specs │
└─────────────────────────────────────────────────────────┘
```

**Rule:** `docs/` holds authoritative technical detail. Memory bank holds distilled operational
state. Skills hold reusable workflows. Root `AGENTS.md` links — does not duplicate.

## Memory-bank tiers (multi-agent safe)

The memory bank is partitioned so that **parallel agents working in separate worktrees never edit
the same file**. See [DEC-014](../.agents/memory-bank/records/decisions/DEC-014.md) for rationale.

| Tier | Edit model | Conflict risk |
|---|---|---|
| **Reference files** (`projectbrief.md`, `systemPatterns.md`, `techContext.md`, `productContext.md` prose) | Hand-edit, low-frequency | Low |
| **Generated indexes** (`activeContext.md`, `progress.md`, `decisions.md`, `productContext.md` tables, `plans/00-index.md`) | Regenerated by `sync-memory-index.sh`; never hand-edit | None (deterministic) |
| **Append-only registries** (`registry/*.jsonl`) | One JSON object per line; mint `max+1`; never edit existing line | Last-line only (trivial) |
| **Per-record files** (`records/progress/M{NN}.md`, `records/active/M{NN}.md`, `records/decisions/DEC-###.md`) | One file per record | None (distinct files) |
| **Module locks** (`locks/<module>.md`) | Create/overwrite before editing coupled pairs | Textual (detectable) |
| **Worktree scratchpads** (`worktrees/<branch-slug>/`, git-ignored) | Per-agent transient drafts | None (never merged) |

## Session workflow

### Start (substantial task)

1. `.agents/memory-bank/projectbrief.md`
2. `.agents/memory-bank/activeContext.md` (generated)
3. `.agents/memory-bank/systemPatterns.md`
4. `.agents/memory-bank/techContext.md`
5. Root `AGENTS.md`
6. Nearest module `AGENTS.md`
7. Load skills per triggers in root AGENTS.md

### During

- TDD + requirement alignment + security-check (see root AGENTS.md)
- Respect Modulith boundaries
- MCP optional invariant (NFR-001)
- Acquire `locks/<module>.md` before editing coupled file pairs

### End (multi-agent safe)

1. Update `records/active/M{NN}.md` (or move to `records/progress/` on completion).
2. Append `registry/*.jsonl` rows for new stable IDs — never edit existing lines.
3. Update `systemPatterns.md`/`techContext.md` if architecture/stack shifted.
4. Acquire/release `locks/<module>.md` for coupled-file edits.
5. Run `scripts/sync-memory-index.sh` to regenerate indexes (CI asserts `--check`).
6. Sync links to `docs/` if canonical docs changed.

## Traceability

Stable IDs (in `registry/*.jsonl`):

| ID type | Registry file | Usage |
|---|---|---|
| `REQ-###` | `req.jsonl` | Functional requirements |
| `NFR-###` | `nfr.jsonl` | Non-functional requirements |
| `SCN-###` | `scn.jsonl` | BDD scenarios |
| `TEST-###` | `test.jsonl` | Test artifacts (class#method) |
| `DEC-###` | `dec.jsonl` | Decisions (body in `records/decisions/DEC-###.md`) |
| `RISK-###` | `risk.jsonl` | Known risks |
| `TASK-###` | `task.jsonl` | Plan tasks |

Milestones **M1–M17** are recorded in `records/progress/M{NN}.md`. Prose traceability tables in
`productContext.md` are generated from the registries; do not hand-edit.

Prefer tables and stable links over prose-only traceability. Use the `bdd-traceability` skill for
requirement → Gherkin → test linkage.

## Adding skills

1. Create `.agents/skills/<kebab-name>/SKILL.md`
2. YAML frontmatter: `name`, `description`, `version`, `tags`
3. Sections: Description, When to use, Instructions, Boundaries
4. Add row to root AGENTS.md Skills index
5. Do **not** duplicate into IDE-specific folders — adapters transform from here

## Maintaining memory bank

| Event | Action |
|---|---|
| Milestone starts | Create `records/active/M{NN}.md`; acquire `locks/<module>.md` |
| Milestone completes | Move record to `records/progress/M{NN}.md`; release lock |
| New requirement/scenario/test/risk/decision/task | Append one line to matching `registry/*.jsonl` |
| Architecture/boundary change | Update `systemPatterns.md`; append `DEC-###` to `dec.jsonl` + `records/decisions/` |
| Stack/tooling change | Update `techContext.md` |
| Doc/code mismatch | Note in `records/active/M{NN}.md`; flag for human review |
| After any registry/record change | Run `scripts/sync-memory-index.sh`; CI runs `--check` |

**Never store:** secrets, large code dumps, raw chat logs, duplicated canonical docs.

## IDE adapters (future)

- Cursor: may symlink or generate rules from `AGENTS.md` + skills
- MCP: optional project tools reading `docs/` and memory bank
- Canonical source remains `.agents/skills` and `.agents/memory-bank`

## Java Cucumber rule

When Cucumber is adopted:

- `.feature` files in acceptance-test layer
- Tags: `@req-NNN`, `@mN`, module tags (`@chat`, `@llm`)
- Thin step definitions; domain logic in services
- Scenario names = business outcomes
- Link scenarios to `SCN-###` IDs in `registry/scn.jsonl`

## Anti-patterns

- Hand-editing a generated index file instead of editing source + re-running `sync-memory-index.sh`
- Editing an existing registry line instead of appending (breaks ID stability)
- Bundling multiple milestones into one record file (reintroduces single-file rewrite conflict)
- Editing coupled files without holding the module lock
- Merging a worktree scratchpad into main
- REQ IDs in docs but not in tests
- Scenarios mirroring screens not behavior
- Nested `AGENTS.md` duplicating root content
- Memory bank copying full SRS
- Stale traceability after refactor
